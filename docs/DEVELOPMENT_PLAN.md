# TechPulse 개발 계획서

## 개요

본 문서는 TechPulse (YouTube 테크 리뷰 트렌드 분석기) 프로젝트의 남은 개발 작업을 정의합니다.
개발자가 각 작업의 배경, 범위, 인수 조건, 의존성을 파악하여 독립적으로 작업할 수 있도록 구성되었습니다.

**참고 문서**: [PRD](./PRD.md)

---

## 현재 상태 (완료된 작업)

| 완료 항목 | 설명 |
|----------|------|
| Phase 1 MVP | FastAPI 백엔드 + Next.js 대시보드 + CSV/Excel 내보내기 |
| 1주차 안정화 | 에러 핸들링, 로깅, Rate Limiting, React Query, ErrorBoundary, CI/CD, Docker |
| 2주차 최적화 | 페이지네이션, GZip 압축, 쿼터 절감, 설정 엔드포인트 |
| 3주차 DB 연동 | Supabase 스키마, 분석 결과 저장, 트렌드 비교 API |

## 남은 작업 목록

### EP-01: 프론트엔드 트렌드 비교 페이지 구현

- **배경**: 백엔드에 `/api/trends` 엔드포인트가 구현되었으나 프론트엔드에 트렌드 비교 UI가 없음. PRD Phase 2의 핵심 기능으로, 마케터가 주간/월간 키워드 변동을 시각적으로 확인해야 함.
- **작업 내용**:
  - `/trends` 페이지 생성 (Next.js App Router)
  - 키워드 순위 변동 테이블 (▲▼ 표시, 신규 진입 하이라이트)
  - 주간 트렌드 라인 차트 (Recharts)
  - 키워드 클릭 시 해당 키워드의 시계열 이력 모달
  - 네비게이션 바에 "트렌드" 링크 추가
- **인수 조건**:
  - [ ] `/trends` 페이지가 정상 렌더링됨
  - [ ] 순위 상승/하락/신규 표시가 올바르게 동작함
  - [ ] 라인 차트에 최소 2주 데이터가 표시됨
  - [ ] 모바일 반응형 레이아웃 동작
  - [ ] TypeScript 빌드 에러 없음
- **의존성**: 없음 (백엔드 API 이미 구현됨)

---

### EP-02: 설정 페이지 구현

- **배경**: PRD에 정의된 `/settings` 페이지가 미구현 상태. 검색어 편집, API 키 관리, 카테고리 설정 등을 GUI에서 관리해야 팀원들이 코드 수정 없이 분석 조건을 변경 가능.
- **작업 내용**:
  - `/settings` 페이지 생성
  - 검색어 목록 편집 UI (한국어/영어 분리, 추가/삭제)
  - 관심사 카테고리 확인 (현재 8개 카테고리 표시)
  - API 키 입력 필드 (마스킹 처리)
  - 캐시 TTL 설정 표시
  - 캐시 수동 초기화 버튼 (`POST /api/cache/clear` 호출)
  - `GET /api/settings` 연동
- **인수 조건**:
  - [ ] `/settings` 페이지가 현재 설정값을 정확히 표시함
  - [ ] 캐시 초기화 버튼이 정상 동작함
  - [ ] API 키가 마스킹되어 표시됨
  - [ ] 네비게이션에서 설정 페이지 접근 가능
- **의존성**: 없음

---

### EP-03: 이메일 리포트 시스템 구현

- **배경**: PRD Phase 3 기능. 마케팅팀이 매주 월요일 자동 리포트를 받아 트렌드를 확인하는 것이 핵심 사용 시나리오. Resend API를 활용한 이메일 발송 시스템이 필요.
- **작업 내용**:
  - Backend: Resend API 연동 (`resend` 패키지)
  - Backend: 이메일 템플릿 (HTML) — 주간 키워드 TOP 10, 관심사 분포, 1위 영상
  - Backend: `/api/report/send` 엔드포인트 (수동 발송)
  - Backend: `/api/report/recipients` 수신자 CRUD
  - Frontend: 리포트 설정 UI (수신자 관리, 미리보기)
  - Vercel Cron: 매주 월요일 09:00 KST 자동 발송 스케줄
- **인수 조건**:
  - [ ] 수동 발송 시 이메일이 정상 도착함
  - [ ] 이메일 내 키워드 테이블과 순위 변동이 포함됨
  - [ ] 수신자 추가/삭제가 정상 동작함
  - [ ] 이메일 미리보기 기능 동작
- **의존성**: EP-01 (트렌드 데이터가 리포트에 포함되므로)

---

### EP-04: NextAuth.js 팀 인증 시스템 구현

- **배경**: PRD Phase 3 기능. 여러 팀원이 대시보드를 공유하려면 인증이 필요. Google OAuth 로그인 + 팀 워크스페이스 개념 도입.
- **작업 내용**:
  - NextAuth.js 설치 및 설정 (Google OAuth provider)
  - Supabase에 users, teams 테이블 추가
  - `/login` 페이지 구현
  - 권한 모델: Admin (설정 변경, 팀원 초대) / Member (조회, 내보내기)
  - 보호된 라우트 미들웨어 (인증 안 된 사용자 리다이렉트)
  - 팀 관리 UI (팀원 목록, 초대, 역할 변경)
- **인수 조건**:
  - [ ] Google OAuth 로그인/로그아웃 정상 동작
  - [ ] 미인증 사용자가 대시보드 접근 시 로그인 페이지로 리다이렉트
  - [ ] Admin이 팀원을 초대하고 역할을 변경할 수 있음
  - [ ] Member는 설정 페이지에서 수정 불가 (읽기만 가능)
- **의존성**: EP-02 (설정 페이지에 권한 적용 필요)

### EP-05: Vercel 배포 파이프라인 구성

- **배경**: 현재 로컬 환경에서만 실행 가능. PRD에서 Vercel 배포를 요구하며, 팀 전체가 접근할 수 있는 프로덕션 환경이 필요.
- **작업 내용**:
  - Frontend: `next.config.mjs`에 standalone 출력 설정
  - Frontend: Vercel 프로젝트 생성 및 환경변수 설정
  - Backend: Vercel Serverless Functions로 마이그레이션 또는 Railway/Render 배포
  - 환경 분리: dev / staging / production
  - 커스텀 도메인 연결 (선택)
  - Vercel Cron Jobs 설정 (매일 06:00 분석 실행)
- **인수 조건**:
  - [ ] 프로덕션 URL에서 대시보드 접근 가능
  - [ ] 환경변수가 안전하게 관리됨 (Vercel 대시보드)
  - [ ] Cron Job이 매일 자동 실행되어 DB에 데이터 저장
  - [ ] HTTPS 적용됨
- **의존성**: EP-04 (인증 적용 후 배포해야 보안 확보)

### EP-06: 접근성(a11y) 및 다크/라이트 모드

- **배경**: 현재 다크 모드만 지원하며, ARIA 레이블이 없어 스크린 리더 사용자가 접근 불가. 웹 접근성 표준(WCAG 2.1 AA) 준수 필요.
- **작업 내용**:
  - 모든 인터랙티브 요소에 `aria-label` 추가
  - 차트 컴포넌트에 대체 텍스트 (데이터 테이블 폴백)
  - 키보드 네비게이션 지원 (Tab, Enter, Escape)
  - 다크/라이트 모드 토글 구현 (Tailwind `dark:` 유틸리티)
  - 사용자 선호 테마 localStorage 저장
  - 색상 대비 비율 4.5:1 이상 확보
- **인수 조건**:
  - [ ] 라이트/다크 모드 전환이 정상 동작
  - [ ] 테마 선택이 새로고침 후에도 유지됨
  - [ ] 모든 버튼/링크에 aria-label 존재
  - [ ] 키보드만으로 전체 네비게이션 가능
  - [ ] Lighthouse Accessibility 점수 90 이상
- **의존성**: 없음 (독립 작업 가능)

### EP-07: 종합 테스트 커버리지 확대

- **배경**: 현재 백엔드 테스트 10개가 존재하나, 프론트엔드 테스트가 없고 통합 테스트도 미구현. 리팩토링 안전성과 회귀 방지를 위해 커버리지 60% 이상 필요.
- **작업 내용**:
  - Backend: database.py 테스트 (Supabase 목 연동)
  - Backend: export 엔드포인트 통합 테스트
  - Frontend: vitest 설정 + React Testing Library
  - Frontend: SummaryCards, VideoTable, RegionToggle 컴포넌트 테스트
  - Frontend: 페이지 렌더링 테스트 (MSW로 API 목킹)
  - E2E: Playwright 기본 시나리오 (대시보드 → 분석 → 내보내기)
- **인수 조건**:
  - [ ] Backend 테스트 커버리지 60% 이상
  - [ ] Frontend 컴포넌트 테스트 최소 5개
  - [ ] E2E 시나리오 1개 이상 통과
  - [ ] CI에서 모든 테스트 자동 실행
- **의존성**: EP-01, EP-02 (새 페이지 구현 후 테스트)

### EP-08: API 문서화 및 개발자 가이드

- **배경**: 현재 API 문서가 README에 간략히만 기술되어 있음. 새 개발자 온보딩과 외부 연동을 위해 체계적 문서 필요.
- **작업 내용**:

  - `docs/CONFIGURATION.md` 작성 (환경변수 전체 목록)
  - `docs/DEPLOYMENT.md` 작성 (Vercel/Docker 배포 가이드)
  - `docs/TROUBLESHOOTING.md` 작성 (FAQ: API 쿼터, CORS 등)

- **인수 조건**:
  - [ ] `/docs` 경로에서 Swagger UI 접근 가능
  - [ ] 모든 엔드포인트에 요청/응답 예시 포함
  - [ ] 환경변수 목록이 최신 상태
  - [ ] 배포 가이드로 신규 개발자가 30분 내 로컬 실행 가능
